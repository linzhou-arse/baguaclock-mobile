# 专业开发规范集

## 规则名称：专业开发规范集
**适用范围：** 全项目代码生成与修改
**优先级：** 高
**强制执行：** 是

## 核心规则

### 1. 数据真实性原则（不做假）
- 生成代码时不得使用虚假数据或占位符
- 所有示例数据必须是合理的真实场景
- 函数返回值必须符合实际业务逻辑
- 禁止使用 `pass`、`TODO` 等空实现

### 2. 质量不降级原则
- 不为快速实现而降低代码质量
- 必须遵循最佳实践和设计模式
- 优先选择可维护性高的解决方案
- 禁止硬编码和魔法数字

### 3. 设计稳定性原则
- 严格按照既定接口和架构实现
- 不得擅自修改已有设计约束
- 保持与现有代码风格一致
- 遵循项目的模块化设计

### 4. 修改前确认原则
- 需求不明确时只提供分析建议
- 修改代码前必须确认具体要求
- 涉及核心逻辑时需要详细说明影响
- 优先询问而非假设需求

### 5. 循环安全原则
- 所有循环必须有明确的退出条件
- 递归函数必须有终止条件检查
- 避免可能导致无限循环的逻辑
- 对迭代次数进行合理限制

### 6. 语法规范原则
- 生成的代码必须通过语法检查
- 遵循 Python PEP8 编码规范
- 变量名使用有意义的英文命名
- 函数和类名遵循驼峰命名法

### 7. 格式一致性原则
- 使用4个空格进行缩进（不使用Tab）
- 行长度不超过88字符
- 在运算符前后添加空格
- 保持空行的合理使用

### 8. 深度问题解决原则
- 必须分析问题的根本原因，不能只修改表面现象
- 制定完整的解决方案，避免头痛医头、脚痛医脚
- 每次修改都要追踪到问题的源头
- 建立问题-原因-方案-验证的完整链路

### 9. 功能完整性原则
- 严禁随意删除或简化原有功能
- 不得因为实现困难而降级功能需求
- 新功能必须完整可用，不能添加半成品功能
- 保持功能与最初设计的一致性

### 10. 代码清理规范
- 增加新功能后必须清理废弃的旧函数
- 删除无用的导入、变量和注释代码
- 更新或删除过时的调用关系
- 保持代码库整洁，避免冗余代码

### 11. 系统性思考原则
- 修改前必须分析对整体架构的影响
- 考虑代码的逻辑一致性和功能匹配性
- 评估与其他模块的兼容性
- 确保修改不破坏系统的整体设计

### 12. 验证完整性原则
- 任何修改完成后必须进行功能验证
- 不能未经测试就声称问题已解决
- 建立修改前后的对比验证机制
- 确保修改效果符合预期目标

## 具体实施标准

### 代码生成时检查清单
- [ ] 无虚假数据或占位符
- [ ] 包含必要的错误处理
- [ ] 有合理的退出条件
- [ ] 符合项目命名规范
- [ ] 通过语法验证
- [ ] 缩进格式正确
- [ ] 功能完整可用，非半成品

### 问题修复时检查清单
- [ ] 已分析问题根本原因
- [ ] 制定了完整解决方案
- [ ] 考虑了对系统其他部分的影响
- [ ] 清理了相关的废弃代码
- [ ] 进行了完整的功能验证
- [ ] 确认问题已真正解决

### 代码修改时检查清单
- [ ] 需求已明确确认
- [ ] 不破坏现有设计
- [ ] 保持接口兼容性
- [ ] 清理了废弃函数和代码
- [ ] 更新了相关调用关系
- [ ] 测试用例已更新
- [ ] 文档同步更新
- [ ] 功能验证完成

### 异常情况处理
- 当规则冲突时，优先保证系统稳定性
- 当性能与规范冲突时，在注释中说明原因
- 紧急修复时可临时放宽规范，但需后续整改
- 遇到重大问题时，必须停止修改并寻求帮助

### 质量保证机制
- 建立修改前的备份机制
- 实施代码审查制度
- 建立问题追踪和解决记录
- 定期进行代码质量审计

## 应用场景
1. **代码生成**：AI辅助生成新代码
2. **代码修改**：重构或功能增强
3. **代码审查**：质量检查和规范验证
4. **团队协作**：统一开发标准

## 执行方式
- 在Qoder中设置为全局规则
- 每次AI辅助时自动应用
- 可以临时禁用特定规则
- 支持项目级别的规则覆盖
